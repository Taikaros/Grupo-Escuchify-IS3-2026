# Steps - Módulo 6: Certificados

| Paso | Sub-paso | Descripción | Archivos afectados | Criterio de completitud |
|------|----------|-------------|-------------------|------------------------|
| **Paso 1** (2 días): Diseño y documentación de entidades y endpoints | | | | |
| 1.1 | Crear tabla `TipoCertificado` en PostgreSQL | Crear tabla con campos: `id_tipo_certificado` (INT PK AUTO_INCREMENT), `nombre` (VARCHAR(50) NOT NULL UNIQUE). Insertar datos seed: 'ASISTENCIA', 'APROBACION', 'PARTICIPACION'. | `src/database/migrations/` (nuevo archivo de migración), `src/database/schema.sql` | La migración se ejecuta sin errores, tabla creada con constraint UNIQUE, datos seed insertados correctamente. |
| 1.2 | Crear tabla `Certificado` en PostgreSQL | Crear tabla con campos: `id_certificado` (INT PK AUTO_INCREMENT), `id_inscripcion` (INT NOT NULL FK→Inscripción), `id_evento` (INT NOT NULL FK→Evento), `id_tipo_certificado` (INT NOT NULL FK→TipoCertificado), `fecha_emision` (DATETIME DEFAULT CURRENT_TIMESTAMP), `calificacion` (DECIMAL(5,2)). Agregar índice en `id_inscripcion` e `id_evento`. | `src/database/migrations/` (nuevo archivo de migración), `src/database/schema.sql` | La migración se ejecuta sin errores, tabla creada con todas las FKs e índices funcionando correctamente. |
| 1.3 | Definir modelo/repositorio de Certificado y TipoCertificado | Crear repositorio con métodos: `crearCertificado()`, `obtenerPorId()`, `obtenerPorInscripcion()`, `obtenerPorEvento()`, `crearTipoCertificado()`, `obtenerTipoPorId()`, `obtenerTipos()`. Consultas parametrizadas. | `src/modules/certificados/repositories/certificadoRepository.js`, `src/modules/certificados/repositories/tipoCertificadoRepository.js` | Todos los métodos funcionan con pruebas unitarias verificando consultas parametrizadas y mapeo correcto. |
| 1.4 | Configurar router y controllers del módulo Certificados | Crear router Express con 3 endpoints: POST `/api/v1/tipos-certificado`, POST `/api/v1/certificados`, GET `/api/v1/certificados/:id/pdf`. Controllers con validación de request y manejo unificado de errores. | `src/modules/certificados/routes/certificadoRoutes.js`, `src/modules/certificados/controllers/certificadoController.js`, `src/modules/certificados/controllers/tipoCertificadoController.js`, `src/app.js` | Los 3 endpoints responden correctamente; rutas registradas bajo `/api/v1/`. |
| **Paso 2** (3 días): Desarrollo de generación de certificados por tipo | | | | |
| 2.1 | Validación de request para POST /api/v1/certificados | Validar que el body contenga `id_inscripcion` (INT) e `id_tipo_certificado` (INT) como requeridos. `calificacion` (DECIMAL) es opcional. Retornar 400 `{ "error": "ERROR_VALIDACION" }` si faltan campos. | `src/modules/certificados/controllers/certificadoController.js`, `src/modules/certificados/validators/certificadoValidator.js` | Requests sin campos requeridos retornan 400. Calificación opcional aceptada. |
| 2.2 | Validación de inscripción (dependencia Módulo 3) | Llamar a `GET /api/v1/inscripciones/{id_inscripcion}` del Módulo 3. Si retorna 404 → 404 `{ "error": "INSCRIPCION_NO_ENCONTRADA" }`. Validar que estado sea 'CONFIRMADA'. | `src/modules/certificados/services/inscripcionService.js` (cliente HTTP al Módulo 3), `src/modules/certificados/controllers/certificadoController.js` | Inscripciones inexistentes retornan 404. Inscripciones canceladas son rechazadas. |
| 2.3 | Validación de acreditación (RN1 - certificado de asistencia) | Si `id_tipo_certificado` corresponde a 'ASISTENCIA', consultar al Módulo 4 (Roles y Acreditación) para verificar que la inscripción tiene acreditación confirmada. Si no está acreditado → 400 `{ "error": "NO_ACREDITADO" }`. | `src/modules/certificados/services/acreditacionService.js` (cliente HTTP al Módulo 4), `src/modules/certificados/controllers/certificadoController.js` | Solo inscripciones acreditadas pueden obtener certificado de asistencia. No acreditado → 400. |
| 2.4 | Validación de calificación mínima (RN2 - certificado de aprobación) | Si `id_tipo_certificado` corresponde a 'APROBACION', validar que `calificacion` esté presente y sea >= calificación mínima definida por el organizador. Si falta o es insuficiente → 400 `{ "error": "CALIFICACION_INSUFICIENTE" }`. Consultar calificación mínima desde Módulo 2 (Evento). | `src/modules/certificados/services/certificadoService.js`, `src/modules/certificados/controllers/certificadoController.js`, `src/modules/certificados/services/eventoService.js` (cliente HTTP al Módulo 2) | Certificados de aprobación requieren calificación >= mínimo. Sin calificación o insuficiente → 400. |
| 2.5 | Validación de rol de disertante (RN3 - certificado de participación) | Si `id_tipo_certificado` corresponde a 'PARTICIPACION', consultar al Módulo 4 para verificar que el usuario de la inscripción tiene rol de 'DISERTANTE' en el evento. Si no tiene el rol → 409 `{ "error": "REQUISITOS_NO_CUMPLIDOS" }`. | `src/modules/certificados/services/rolService.js` (cliente HTTP al Módulo 4), `src/modules/certificados/controllers/certificadoController.js` | Solo usuarios con rol disertante obtienen certificado de participación. Sin rol → 409. |
| 2.6 | Validar que no exista certificado duplicado | Verificar si ya existe un certificado con la misma combinación `(id_inscripcion, id_tipo_certificado)`. Si existe → 409 `{ "error": "CERTIFICADO_YA_EXISTENTE" }`. | `src/modules/certificados/repositories/certificadoRepository.js`, `src/modules/certificados/services/certificadoService.js` | No se permite generar el mismo tipo de certificado dos veces para la misma inscripción. |
| 2.7 | Crear certificado con transacción | Dentro de transacción de BD, insertar certificado con `id_inscripcion`, `id_evento`, `id_tipo_certificado`, `fecha_emision = CURRENT_TIMESTAMP`, `calificacion` (si aplica). Confirmar si OK, rollback si error. Retornar 201 `{ "id_certificado": "{{id}}" }`. | `src/modules/certificados/repositories/certificadoRepository.js`, `src/modules/certificados/services/certificadoService.js` | Certificado creado retorna 201. Rollback automático si falla algún paso. |
| **Paso 3** (2 días): Desarrollo de exportación a PDF | | | | |
| 3.1 | Configurar librería de generación de PDF | Instalar y configurar librería PDF (ej: pdfkit, puppeteer). Configurar template base del certificado con placeholders: nombre participante, nombre evento, fecha, tipo certificado, calificación (si aplica). | `package.json`, `src/modules/certificados/services/pdfGenerator.js`, `src/modules/certificados/templates/certificadoTemplate.html` | Librería instalada y template funcional con placeholders renderizados correctamente. |
| 3.2 | Implementar route handler GET /api/v1/certificados/:id/pdf | Buscar certificado por `id_certificado`. Si no existe → 404. Obtener datos del participante (Módulo 1), evento (Módulo 2), tipo certificado e inscripción. Generar PDF con los datos. Retornar archivo PDF binario con `Content-Type: application/pdf`. | `src/modules/certificados/controllers/certificadoController.js`, `src/modules/certificados/services/pdfGenerator.js`, `src/modules/certificados/services/usuarioService.js`, `src/modules/certificados/services/eventoService.js` | Certificado existente retorna PDF binario 200 con datos correctos. Inexistente → 404. |
| 3.3 | Implementar formato de PDF por tipo de certificado | Adaptar el contenido del PDF según tipo: ASISTENCIA (nombre, evento, fecha), APROBACION (nombre, evento, fecha, calificación), PARTICIPACION (nombre, evento, fecha, rol disertante). Incluir logos y firma del organizador. | `src/modules/certificados/services/pdfGenerator.js`, `src/modules/certificados/templates/` (templates por tipo) | Cada tipo de certificado genera un PDF con formato y contenido diferenciado correcto. |
| 3.4 | Implementar POST /api/v1/tipos-certificado | Validar `nombre` (VARCHAR(50)) en el request. Verificar que no exista tipo duplicado (409 `{ "error": "NOMBRE_DUPLICADO" }`). Insertar en BD. Retornar 201 `{ "id_tipo_certificado", "nombre" }`. | `src/modules/certificados/controllers/tipoCertificadoController.js`, `src/modules/certificados/validators/tipoCertificadoValidator.js` | Tipo creado retorna 201. Nombre duplicado → 409. Validación de campos → 400. |

Mocks a Módulos 1, 2, 3, 4. | `src/modules/certificados/tests/certificadoController.test.js`, `src/modules/certificados/tests/certificadoService.test.js` | Todos los tests pasan. Cobertura de las ramas de error del controller y service. |
| 4.2 | Tests unitarios de generación de PDF | Tests: generar PDF de certificado de asistencia → verifica contenido correcto, generar PDF de aprobación → incluye calificación, generar PDF de participación → incluye rol disertante, certificado inexistente → 404. | `src/modules/certificados/tests/pdfGenerator.test.js`, `src/modules/certificados/tests/certificadoController.test.js` | Todos los tests de PDF pasan. Contenido validado según tipo de certificado. |
| 4.3 | Tests unitarios de tipos de certificado | Tests: crear tipo con nombre válido → 201, crear tipo con nombre duplicado → 409, crear tipo sin nombre → 400, listar tipos → retorna array. | `src/modules/certificados/tests/tipoCertificadoController.test.js` | Todos los 4 tests pasan. BD de test con datos limpios por cada test. |
| 4.5 | Tests de aceptación (E2E) | Tests E2E: 1) Organizador genera certificado de asistencia para acreditado → PDF válido, 2) Generar certificado de aprobación con calificación suficiente → 201, 3) Disertante genera certificado de participación → 201 y PDF correcto. Módulos mockeados. | `src/modules/certificados/tests/e2e/certificadoE2E.test.js` | Los 3 flujos E2E pasan end-to-end. PDFs generados con datos correctos. |
