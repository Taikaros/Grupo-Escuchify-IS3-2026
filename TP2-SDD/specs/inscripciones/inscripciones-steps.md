# Steps - Módulo 3: Inscripciones

| Paso | Sub-paso | Descripción | Archivos afectados | Criterio de completitud |
|------|----------|-------------|-------------------|------------------------|
| **Paso 1** (2 días): Diseño y documentación de entidades y endpoints | | | | |
| 1.1 | Crear tabla `Inscripción` en PostgreSQL | Crear tabla con campos: `id_inscripcion` (INT PK AUTO_INCREMENT), `id_usuario` (INT NOT NULL FK→Usuario), `id_evento` (INT NOT NULL FK→Evento), `fecha_inscripcion` (DATETIME DEFAULT CURRENT_TIMESTAMP), `estado` (VARCHAR(20) DEFAULT 'CONFIRMADA'). Agregar índice en `id_evento` y constraint UNIQUE en `(id_usuario, id_evento)` para RN4. | `src/database/migrations/` (nuevo archivo de migración), `src/database/schema.sql` | La migración se ejecuta sin errores, la tabla existe con todas las columnas, FKs y constraint UNIQUE funcionando correctamente. |
| 1.2 | Definir modelo/repositorio de Inscripción | Crear el repositorio de `Inscripción` con métodos: `crear()`, `obtenerPorId()`, `obtenerPorUsuarioYEvento()`, `contarConfirmadasPorEvento()`, `cancelar()`. Consultas parametrizadas para prevenir SQL injection. | `src/modules/inscripciones/repositories/inscripcionRepository.js` | Todos los métodos del repositorio funcionan con pruebas unitarias que verifican consultas parametrizadas y mapeo correcto de resultados. |
| 1.3 | Configurar router y controllers | Crear router Express bajo `/api/v1/inscripciones` con 4 routes: POST `/`, POST `/manual`, GET `/:id_inscripcion`, DELETE `/:id_inscripcion`. Controllers con validación de request y manejo unificado de errores `{ "error": "CODIGO", "message": "descripcion" }`. | `src/modules/inscripciones/routes/inscripcionRoutes.js`, `src/modules/inscripciones/controllers/inscripcionController.js`, `src/app.js` | Los 4 endpoints responden correctamente; las rutas están registradas y accesibles bajo el prefijo `/api/v1/`. |
| **Paso 2** (2 días): Desarrollo de inscripción autónoma | | | | |
| 2.1 | Validación de request para POST /api/v1/inscripciones | Validar que el body contenga `id_usuario` (INT) e `id_evento` (INT) como campos requeridos. Retornar 400 con `{ "error": "ERROR_VALIDACION" }` si faltan campos o tienen tipo incorrecto. | `src/modules/inscripciones/controllers/inscripcionController.js`, `src/modules/inscripciones/validators/inscripcionValidator.js` | Requests sin campos o con tipos inválidos retornan 400 con el código de error correspondiente. |
| 2.2 | Validación de usuario autenticado | Llamar a `GET /api/v1/usuarios/{id_usuario}` del Módulo 1. Si retorna 404 → 400 `{ "error": "USUARIO_NO_ENCONTRADO" }`. Si `activo = false` → 400 `{ "error": "USUARIO_INACTIVO" }`. | `src/modules/inscripciones/services/usuarioService.js`, `src/modules/inscripciones/controllers/inscripcionController.js` | Inscripciones con usuarios inexistentes o inactivos son rechazadas. Usuarios válidos pasan la validación. |
| 2.3 | Validación de evento, cupo y fecha límite | Llamar a `GET /api/v1/eventos/{id_evento}` del Módulo 2. Validar estado "ACTIVO" (sino → 400 EVENTO_NO_ACTIVO). RN2: si `fecha_actual > fecha_limite_inscripcion` → 409 FECHA_LIMITE_EXPIRADA. RN1: si `count >= cupo_maximo` → 409 CUPO_LLENO. RN4: si ya existe `(id_usuario, id_evento)` → 409 YA_INSCRITO. | `src/modules/inscripciones/services/eventoService.js`, `src/modules/inscripciones/services/inscripcionService.js`, `src/modules/inscripciones/controllers/inscripcionController.js` | Cada regla de negocio (RN1, RN2, RN4) retorna el código de error correcto cuando se viola. Todas las validaciones pasan cuando los datos son correctos. |
| 2.4 | Creación de inscripción con transacción | Dentro de transacción de BD, insertar inscripción con `estado = 'CONFIRMADA'`, `fecha_inscripcion = CURRENT_TIMESTAMP`. Confirmar si OK, rollback si error. Retornar 201 `{ "id_inscripcion": "{{id}}", "estado": "CONFIRMADA" }`. | `src/modules/inscripciones/repositories/inscripcionRepository.js`, `src/modules/inscripciones/services/inscripcionService.js` | Inscripción creada retorna 201. Si falla algún paso, rollback automático. No se persisten datos parciales. |
| **Paso 3** (2 días): Desarrollo de inscripción manual por personal | | | | |
| 3.1 | Validación de request para POST /api/v1/inscripciones/manual | Validar que el body contenga `id_usuario` (INT), `id_evento` (INT) e `id_personal` (INT). Retornar 400 con `{ "error": "ERROR_VALIDACION" }` si faltan campos. | `src/modules/inscripciones/validators/inscripcionValidator.js`, `src/modules/inscripciones/controllers/inscripcionController.js` | Requests sin campos requeridos retornan 400 con código de error. |
| 3.2 | Validación de autorización del personal | Verificar que `id_personal` tenga rol de organizador/personal autorizado para `id_evento`. Consultar Módulo 4 (Roles). Si no tiene permisos → 403 `{ "error": "NO_AUTORIZADO" }`. | `src/modules/inscripciones/services/rolService.js`, `src/modules/inscripciones/controllers/inscripcionController.js` | Solo personal con rol autorizado puede realizar inscripciones manuales. Sin rol → 403. |
| 3.3 | Validaciones de negocio para inscripción manual | Reutilizar validaciones del paso 2.3: evento activo, RN1 (cupo), RN2 (fecha límite), RN4 (duplicado). | `src/modules/inscripciones/services/inscripcionService.js`, `src/modules/inscripciones/controllers/inscripcionController.js` | Inscripción manual aplica todas las reglas de negocio. Cupo lleno → 409, fecha vencida → 409, duplicado → 409. |
| 3.4 | Creación de inscripción manual con transacción | Dentro de transacción, insertar inscripción con mismos campos que autónoma. Registrar `id_personal` en log de auditoría. Retornar 201 `{ "id_inscripcion": "{{id}}", "estado": "CONFIRMADA" }`. | `src/modules/inscripciones/repositories/inscripcionRepository.js`, `src/modules/inscripciones/services/inscripcionService.js` | Inscripción manual retorna 201. Log de auditoría registra quién realizó la inscripción. Rollback en caso de error. |
| **Paso 4** (1 día): Desarrollo de cancelación de inscripciones | | | | |
| 4.1 | Route handler GET /api/v1/inscripciones/{id} | Buscar inscripción por `id_inscripcion`. Si no existe → 404 `{ "error": "NO_ENCONTRADO" }`. Si existe → 200 con `{ "id_inscripcion", "id_usuario", "id_evento", "estado" }`. | `src/modules/inscripciones/controllers/inscripcionController.js`, `src/modules/inscripciones/repositories/inscripcionRepository.js` | Inscripción existente retorna 200. Inscripción inexistente retorna 404. |
| 4.2 | Validación de autorización para cancelación | Verificar que el usuario autenticado sea el mismo que `id_usuario` de la inscripción. Si no coincide → 403 `{ "error": "NO_AUTORIZADO" }`. | `src/modules/inscripciones/controllers/inscripcionController.js` | Solo el usuario dueño puede cancelar. Otro usuario → 403. |
| 4.3 | Validación de regla de 24 horas (RN3) | Obtener `fecha_inicio` del evento desde Módulo 2. Calcular diferencia entre `fecha_actual` y `fecha_inicio`. Si `< 24 horas` → 400 `{ "error": "CANCELACION_NO_PERMITIDA" }`. | `src/modules/inscripciones/services/eventoService.js`, `src/modules/inscripciones/controllers/inscripcionController.js` | Cancelación con menos de 24h retorna 400. Con más de 24h pasa. |
| 4.4 | Cancelación de inscripción | Actualizar `estado` a 'CANCELADA'. El cupo se libera automáticamente (conteo excluye canceladas). Retornar 204 (sin contenido). | `src/modules/inscripciones/repositories/inscripcionRepository.js`, `src/modules/inscripciones/services/inscripcionService.js` | Estado 'CANCELADA' en BD. Endpoint retorna 204. Conteo de cupos refleja la liberación. |
| **Paso 5** (2 días): Pruebas unitarias y de integración | | | | |
| 5.1 | Tests unitarios de validaciones de inscripción | Tests: cupo lleno → 409 CUPO_LLENO, fecha expirada → 409 FECHA_LIMITE_EXPIRADA, duplicado → 409 YA_INSCRITO, evento no activo → 400, usuario inexistente → 400, evento inexistente → 400. Mocks a Módulo 1 y 2. | `src/modules/inscripciones/tests/inscripcionController.test.js`, `src/modules/inscripciones/tests/inscripcionService.test.js` | Todos los 6 tests pasan. Cobertura de las ramas de error del controller y service. |
| 5.2 | Tests unitarios de cancelación | Tests: cancelar 12h antes → 400 CANCELACION_NO_PERMITIDA, cancelar 48h antes → 204, cancelar inexistente → 404, cancelar de otro usuario → 403 NO_AUTORIZADO. Mock de fecha. | `src/modules/inscripciones/tests/inscripcionController.test.js` | Todos los 4 tests pasan. Mock de fecha configurado correctamente. |
| 5.3 | Tests unitarios de inscripción manual | Tests: personal sin rol → 403 NO_AUTORIZADO, personal autorizado → 201, cupo lleno → 409 CUPO_LLENO. Mock Módulo 4. | `src/modules/inscripciones/tests/inscripcionController.test.js` | Todos los 3 tests pasan. Mock de autorización configurado correctamente. |
| 5.4 | Tests de integración con BD | Tests: inscripción persiste 'CONFIRMADA', cancelación actualiza 'CANCELADA', transacción rollback al fallar, FK con usuario inexistente falla, FK con evento inexistente falla, UNIQUE con duplicado falla. | `src/modules/inscripciones/tests/integration/inscripcionIntegration.test.js`, configuración BD de test | Todos los 6 tests pasan. BD de test con datos limpios por cada test. |
| 5.5 | Tests de aceptación (E2E) | Tests E2E: 1) Usuario se inscribe → confirmación, 2) Usuario cancela 3 días antes → cupo liberado, 3) Personal inscribe manualmente → confirmación. Módulos mockeados. | `src/modules/inscripciones/tests/e2e/inscripcionE2E.test.js` | Los 3 flujos E2E pasan end-to-end. Datos en BD reflejan cada estado correctamente. |
